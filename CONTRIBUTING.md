# Contributing

Keep contributions small, practical, and aligned with the suite's split of responsibilities.

## Before you change anything

- Check whether the change belongs in an existing skill instead of creating overlap.
- Update the trigger test matrix if you change frontmatter wording.
- Update the suite index if you change role boundaries.
- Update the workflow file if you change the recommended flow.

## What to include in changes

- concise SKILL.md wording
- reusable references when needed
- example requests only when they clarify triggers
- validation notes if they help future maintenance

## What to avoid

- broadening a skill until it overlaps another one
- adding documentation that only repeats existing files
- merging skills just because they feel related

## Validation

If a skill pack changes, validate it before publishing.
If the suite changes, keep the README short and the workflow accurate.

## Style

- favor clarity over cleverness
- keep dependencies low
- preserve the four-way split: inspect, manifest, risk, plan
