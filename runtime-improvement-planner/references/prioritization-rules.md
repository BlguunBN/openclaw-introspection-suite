# Prioritization Rules

Use this file when deciding what should happen now versus later.

## Scoring dimensions

Use lightweight scoring, not fake precision.

### 1. User value
Ask:
- Does this make the setup more useful in normal daily use?
- Does it answer a recurring pain point?
- Does it improve clarity or trust?

### 2. Safety gain
Ask:
- Does this reduce broad trust or unclear behavior?
- Does it improve approval boundaries?
- Does it reduce external-risk exposure?

### 3. Effort
Ask:
- How long will this take to build?
- How annoying will it be to maintain?
- Does it create another thing to remember?

### 4. Token efficiency
Ask:
- Will this reduce repeated explanation or repeated analysis?
- Will it help choose the right path faster?
- Will it cut context bloat?

### 5. Complexity cost
Ask:
- Does this add another layer, abstraction, or skill that overlaps existing ones?
- Does it make behavior harder to predict?

## Bucket rules

### Do now
Choose items that are:
- high user value or high safety gain
- low or moderate effort
- low complexity cost

### Do next
Choose items that are:
- clearly worthwhile
- but not the single best immediate move

### Nice to have
Choose items that are:
- useful but not urgent
- lower leverage
- more cosmetic or comfort-oriented

### Not worth it right now
Choose items that are:
- duplicative
- clever but low value
- too expensive for the payoff
- likely to confuse future use

## Common prioritization mistakes

### Collecting skills instead of solving problems
Do not recommend another skill just because it fits the pattern.

### Overvaluing elegance
A perfectly structured setup is not always the most useful setup.

### Ignoring maintenance burden
A new capability that must be constantly explained or managed is more expensive than it looks.
